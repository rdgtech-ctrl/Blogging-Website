### React - Router - Dom
React Router DOM lets you build multiple pages in a React app without actually reloading the browser.

Without React Router DOM:
Your app only has one page — everything is on the same screen.
With React Router DOM:
You can have different URLs showing different components, like:
/          → Home page
/login     → Login page
/register  → Register page
/profile   → Profile page
```
import { BrowserRouter, Routes, Route } from "react-router-dom"
import Home from "./pages/Home"
import Login from "./pages/Login"

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/login" element={<Login />} />
      </Routes>
    </BrowserRouter>
  )
}
```

BrowserRouter — wraps your whole app, enables routing
Routes — container for all your routes
Route — maps a URL path to a component
Link — like <a> tag but without page reload
useNavigate — lets you redirect the user in code (e.g. after login)
useParams — lets you read URL parameters like /post/123

### cors
Your frontend runs on http://localhost:5173 and your backend runs on http://localhost:8000. These are different origins (different ports = different origin).
By default, browsers block requests from one origin to another for security reasons.So when your React app tries to call your Express API,the browser will block it

With cors package you fix it:
javascriptimport cors from 'cors'

app.use(cors({
  origin: 'http://localhost:5173',  // allow your frontend
  credentials: true                  // allow cookies to be sent
}))
Now your backend says "I trust requests coming from localhost:5173" and the browser allows it.

credentials: true is important because you're using cookies (your JWT token is stored in a cookie). Without it, cookies won't be sent along with requests.

In short — cors is needed whenever your frontend and backend are on different ports or domains.

### why inside redux we are using files named in .js not in .jsx


Because Redux files contain pure JavaScript logic — no JSX (no HTML-like syntax) at all.

.jsx is only needed when a file contains JSX syntax like:
jsxreturn (
  <div>
    <h1>Hello</h1>
  </div>
)

Redux files like store.js or authSlice.js only contain plain JavaScript:
javascriptimport { createSlice } from '@reduxjs/toolkit'

const authSlice = createSlice({
  name: 'auth',
  initialState: { user: null },
  reducers: {
    setUser: (state, action) => {
      state.user = action.payload
    }
  }
})
No HTML, no components, no JSX — just plain JS objects and functions.

Simple rule:
File containsExtensionJSX (HTML-like syntax).jsxPure JavaScript logic.js
So components like Login.jsx, Signup.jsx, Navbar.jsx use .jsx because they return UI. But store.js, authSlice.js use .js because they only contain logic.

###### Redux Toolkit
### authSlice.js


Manages who is logged in and loading state.
```
state.auth.user    → the logged in user object (or null)
state.auth.loading → true/false (show spinner or not)
```
Any component can read or update this — that's why Login.jsx uses:

```
useSelector(store => store.auth)       // read loading, user
dispatch(setLoading(true))             // update loading
dispatch(setUser(res.data.user))       // update user
```


### themeSlice.js
Manages light or dark mode.
```
state.theme.theme -> "light" or "dark"
```
Any component (like a toggle button in Navbar) can do:
```
dispatch(toggleTheme()) //switches between light and dark
```
### store.js
Combines both slices into one central store:
```
store
 ├── state.auth.loading
 ├── state.auth.user
 └── state.theme.theme
```
### ThemeProvider.jsx
Reads state.theme.theme and puts "light" or "dark" as a className on the outermost div. This is what activates Tailwind's dark: classes throughout your entire app.
```
<div className="dark">          ← when theme is "dark"
  <div className="dark:bg-[rgb(16,23,42)]">   ← this activates ✅
```
